# 런타임 시각화 공백 이슈 분석 리포트

## 이슈 요약
- 증상: **코드는 실행되지만 시각화 영역(`h-full w-full rounded-md bg-muted/5`)이 빈 상태**
- 발생 위치: Graph/Trie/Algorithms 등 **Cytoscape 기반 시각화 전환 이후**
- Worker 로그: `BATCH_STEPS` 정상 수신 (코드 실행은 성공)
- 재현: **첫 실행은 보이나 재실행 후 빈 화면으로 고정**되는 케이스 발생

---

## 원인 가설(우선순위)
1) **Cytoscape 레이아웃 실행 타이밍/컨테이너 크기 0**
   - 컨테이너 크기가 0인 상태에서 레이아웃을 실행하면 좌표 계산 실패 → 빈 화면 유지
   - 이미 ResizeObserver + 재시도 로직 추가했으나, 추가 검증 필요
2) **Adapter 결과가 비어 있음**
   - GraphAdapter/TrieAdapter 결과가 빈 배열이면 시각화가 없게 됨
   - 초기 코드/변수명 규약 위반 시 발생 가능
3) **Visualizer 초기화/타이밍 문제**
   - `cyRef`가 초기화되지 않으면 canvas가 생성되지 않음
   - 초기화 이전에 data effect가 먼저 실행되면 요소가 추가되지 않을 수 있음
4) **Skulpt 미지원 모듈 사용**
   - `heapq` 등 Skulpt 미구현 모듈 사용 시 실행 오류 → 시각화 단계 미진행
5) **재실행 시 상태 초기화 불완전**
   - 기존 cy 인스턴스 상태가 남아 재실행 시 렌더 실패 가능

---

## 확인 방법(런타임 디버그)
1) 콘솔 확인
   - `[Skulpt] Message from Worker: BATCH_STEPS` → 코드 실행 OK
   - `[Cytoscape] layout ...` 로그로 nodes/edges/컨테이너 크기 확인 (dev 전용 로그 추가됨)
2) Cytoscape 요소 수
   - nodes=0 / edges=0이면 Adapter 결과가 비어 있음
3) 컨테이너 크기
   - width/height가 0이면 레이아웃 실패 가능
4) Skulpt 에러 확인
   - `NotImplementedError: heapq is not yet implemented` → 초기 코드 수정 필요
5) 재실행 직후 상태 확인
   - `cy.elements().length` 확인 (0이면 Adapter/주입 문제)

## DFS/BFS 알고리즘 시뮬레이션 현황
### 현재 상태
- **모드**: 전 챕터 Code 기반(트레이스 + 그래프 시각화)
- **Visualizer**
  - Graph 기반: `GraphSvgVisualizer` (traceOnly)
  - Grid 기반: `ArrayGraphVisualizer`
- **Trace 이벤트**: node_active/node_visit/edge_consider/dist_update 중심
- **출력 피드백**: print 표준화 적용(변수 덤프)

### 세부 현황(확장 챕터 포함)
#### DFS
- Basics / Backtracking / Tree Traversal / Cycle Detection / Path Reconstruction
- 공통 구조: `graph` + trace 기반 시각화
- 장점: 방문 순서/상태를 직관적으로 확인 가능
- 한계: 사용자 커스텀 코드에서 trace 규약이 깨지면 시각화 미반영

#### BFS
- Basics / Grid Traversal / Multi-Source / 0-1 BFS / Path Reconstruction
- Grid는 2D 배열 시각화에 최적화(좌표/방문 표시)
- 0-1 BFS는 가중치 정보가 라벨로만 표시되어 **경로 선택 이유 설명이 약함**

### 향후 방향성(정확성 + 학습효과 강화)
1) **난이도 단계별 예제 데이터셋**
   - 각 챕터에 쉬운/중간/실전형 그래프/격자 프리셋 제공
2) **Trace 가이드 템플릿 제공**
   - `trace()` 사용 예시를 코드 상단에 주석으로 고정
3) **0-1 BFS 시각화 강화**
   - 0/1 가중치 간선을 색으로 구분
   - 덱 앞/뒤 삽입 시각화를 별도 패널로 표시
4) **경로 복원 단계 텍스트 추가**
   - “parent가 채워지는 순간 → 역추적” 단계별 안내

### 적용 현황
- DFS/BFS 전 확장 챕터에 **개념별 고정 예제 그래프/격자** 적용
- 난이도 선택 로직은 코드 내부에서 제거하고, **개념에 맞는 대표 사례**만 유지
- 문제 리스트에 난이도 단계 흐름을 반영해 재정렬

## 그래프 배치 연구(다각형 + 중앙 노드 방식)
### 아이디어 요약
- 노드 수가 N일 때 **N-1개를 정다각형 외곽에 배치 + 1개를 중앙에 배치**.
- 예: N=5 → 사각형 + 중앙 1개, N=10 → 9각형 + 중앙 1개.

### 기대 효과(학습 관점)
- **한눈에 노드 수와 간선 분포를 파악**하기 쉬움(균일한 시야 확보).
- 초급 학습에서 “노드=점, 간선=선” 구조를 **빠르게 인지**.
- 노드 간 간격이 균일해 **겹침/밀집** 문제를 줄임.

### 주요 한계/위험
- **중앙 노드가 실제 의미를 갖지 않는 경우** 오해 유발(허브가 아닌데 중심으로 보임).
- 구조적 성질(거리, 군집, 연결성)을 **시각적으로 왜곡**할 수 있음.
- 간선이 많은 그래프는 중앙을 가로지르는 선이 겹쳐 **가독성 하락**.
- DFS/BFS 같은 알고리즘 추적 시, **탐색 트리 구조**가 잘 드러나지 않음.

### 대안/보완 전략
1) **조건부 다각형 레이아웃**
   - 허브 존재 시에만 적용  
   - 조건 예: `max_degree >= 0.5 * (N-1)` 또는 `rootId`가 명확할 때
2) **기본은 원형(Circular) + 정렬 고정**
   - 중앙 노드 없이 **순수 원형 배치**로 오해 최소화
   - 노드 ID 정렬 기반으로 **재실행 시 위치 고정**
3) **상황별 레이아웃 선택**
   - 그래프 일반: 원형 / 밀집 그래프: force-lite  
   - 탐색 알고리즘(DFS/BFS): **BFS 계층형**  
   - MST/Tree: **계층형 트리 배치**
   - Grid 기반 예제: **격자 배치**

### 적용 제안(CTP 기준)
- 기본값: **원형 레이아웃**
- **중앙 노드 레이아웃은 ‘허브 그래프’ 예제에만 적용**
- 학습 목적의 “탐색/경로 강조”가 필요한 알고리즘은 **계층형/순서형 레이아웃** 유지

### 결론
- 다각형+중앙 방식은 **초급 개념 소개에는 유효**하지만,
  **모든 그래프에 적용하면 오해 위험이 높음**.
- 따라서 **조건부(허브/루트 기반) 적용**이 최적이며,
  그 외는 원형/계층형 레이아웃으로 분리 운용해야 함.

## 정다각형 레이아웃 구체안(실행 설계)
### 1) 적용 조건(자동 선택 규칙)
- 기본값: **원형(Circular)**
- 정다각형+중앙 적용 조건(둘 중 하나 충족)
  - `rootId`가 지정되어 있고, 그 노드를 **중앙에 고정**하고 싶을 때
  - 허브 조건: `max_degree >= 0.5 * (N - 1)` (연결 집중 그래프)

### 2) 좌표 산출(정다각형 외곽 + 중앙)
- 노드 수 `N`:
  - `N <= 2`: 일직선 배치 (좌/우 또는 상/하)
  - `N >= 3`:
    - 중앙 노드 1개 지정 (root 또는 max-degree)
    - 외곽 노드 수 `M = N - 1`
    - 반지름 `R = min(width, height) * 0.35`
    - 각도 간격 `theta = 2π / M`
    - 외곽 노드 좌표:
      - `x = cx + R * cos(theta * i - π/2)`
      - `y = cy + R * sin(theta * i - π/2)`
    - 중앙 노드 좌표: `(cx, cy)`

### 3) 노드 배치 순서(안정성)
- 외곽 노드는 **id 정렬** 또는 **degree 내림차순**으로 배치
- 그래야 재실행 시 위치가 바뀌지 않고 **시각화 안정성 확보**

### 4) 간선 표시 규칙
- 중앙 노드가 hub일 때 **허브-외곽 간선 강조**
- 비허브 그래프라면 중앙 고정 대신 **순수 원형 배치**로 회귀

### 5) 적용 예시
- N=5: 4각형 + 중앙 1개
- N=10: 9각형 + 중앙 1개
- N=3: 2각형(라인) + 중앙 1개 → **시각 혼동** 가능하므로
  - N<=4는 **중앙 배치 비활성화** 권장

### 6) 리스크 회피 가드
- MST/DFS/BFS 시각화는 **계층형/순서형 레이아웃 우선**
- 정다각형은 **그래프 구조 소개(초급)**에서만 사용

### 7) 구현 위치(CTP)
- 대상 파일: `web/components/features/ctp/playground/visualizers/graph/graph-svg-visualizer.tsx`
- 신규 옵션 제안: `layoutMode="polygon"`  
  - 조건부 활성화 로직을 내부에서 자동 선택 or registry에서 명시

## 다각형 레이아웃 적용 검토(그래프 시각화 난해도 완화)
### 적용 취지
- 현재 그래프가 **직렬/삼각형 형태로만 반복**되며,
  간선 교차/군집이 많을 때 구조 파악이 어렵다.
- 노드 수가 늘어날수록 “중앙 집중/겹침” 문제가 커진다.

### 제안: 정다각형 배치 기반 구조화
- 노드를 **원형(정다각형)으로 고정 배치**하면
  - 노드 수/연결 상태를 한눈에 파악 가능
  - 재실행 시 위치가 고정되어 학습 안정성 상승
- 허브/루트가 있는 경우에만 **중앙 노드**를 배치해 강조

### 적용 범위(CTP 운영안)
- Graph 개념(Representation/DFS/BFS): **원형 or 다각형 배치 기본**
- 알고리즘(Shortest Path/MST/Topo): **계층형/순서형 유지**
- 허브 그래프 예제(중앙 집중형): **다각형+중앙 노드 모드**

### 기대 효과
- 탐색 순서, 방문 상태 색상 변화가 더 잘 보임
- 작은 그래프에서도 구조가 규칙적으로 보여 **학습 부하 감소**
- 고정 배치로 “코드 실행마다 노드 위치가 바뀌는 문제” 해소

### 리스크/주의
- 일반 그래프를 중앙 허브처럼 보여 **구조적 오해** 유발 가능
- 트리/계층 구조를 다각형으로 배치하면 **부모-자식 관계 가독성 저하**

### 결론
- 다각형 배치는 **그래프 개념 학습(초급)**에는 유효하지만,
  알고리즘 심화(최단경로/MST)에서는 계층형 유지가 적합.

## 방문 기록/상태 패널 도입 필요성(학습 효율 개선)
### 문제 인식
- 현재 그래프 시각화는 **노드/간선 색상 변화** 중심으로만 진행됨
- 방문 순서/큐·스택 상태/거리·부모 정보가 **시각화에 남지 않아**
  학습자가 “왜 이렇게 방문했는지”를 이해하기 어려움

### 개선 방향(필수 구성 요소)
1) **방문 순서(Traversal Order) 패널**
   - 방문한 노드를 리스트로 누적 표기
   - DFS는 `order`, BFS는 `order + dist` 함께 표시
2) **큐/스택 상태 패널**
   - BFS: `queue`/`head` 상태를 시각적으로 표현
   - DFS: `stack` 또는 `call_stack` 상태 표시
3) **거리/부모 테이블**
   - `dist`, `parent` 배열을 별도 표로 출력
   - 현재 노드의 dist/parent를 강조
4) **이벤트 로그 타임라인**
   - `trace()` 이벤트를 텍스트 로그로 누적
   - “node_active → edge_consider → dist_update” 흐름을 기록

### 구현 제안(CTP 기준)
- `CTPPlayground` 우측에 **상태 패널 영역** 추가
- 기존 stdout(터미널)과 분리하여 **학습용 요약 패널** 제공
- 데이터는 다음 규약으로 연결
  - DFS: `order`, `stack`, `parent`, `visited_nodes`
  - BFS: `queue`, `dist`, `parent`, `order`
  - Grid: `frontier_cells`, `dist`, `visited`

### 기대 효과
- “방문 순서”와 “구조 변화”가 동시에 보이므로 **학습 이해도 상승**
- 코딩테스트 대비 시 **큐/스택 상태 추적 능력 강화**

### 적용 현황(이번 단계)
- `CTPPlayground`에 **상태 요약 패널** 추가 완료
- 변수 스냅샷(`variables`)을 step에 포함해 패널에서 읽도록 연결
- 방문 순서/큐/스택/거리/부모/경로를 자동 노출
- 그래프 개념 챕터에 **정다각형 레이아웃** 적용(Representation/DFS/BFS/Cycle)
- 그래프 기본 예제 노드 수를 확대(DFS/BFS/Cycle/Dijkstra/MST)하여 구조 인지 강화
- Graph 개념 챕터는 **그래프 전체 구조를 항상 렌더링**하도록 traceOnly 해제(깜빡임 완화)
- MST 챕터도 **정다각형 레이아웃** 적용(요청 반영)
- Algorithms DFS/BFS(그래프 기반)도 **정다각형 레이아웃** 적용

## 상태 패널 부분 적용 전략(시각화 가림 문제 대응)
### 문제
- 전 챕터에 상태 패널을 동일하게 노출하면 **시각화 영역이 좁아져 학습 흐름을 방해**
- 특히 그래프/정렬/트리 시각화는 **공간 비중이 중요**

### 전략
1) **챕터별 조건부 표시**
   - DFS/BFS/경로복원/0-1 BFS 등 “상태 추적이 핵심”인 챕터에서만 표시
   - 구조 이해 중심(Representation/Trie/Heap) 등은 **기본 숨김**
2) **패널 위치/크기 정책**
   - 기본은 **접기(collapsible) 상태** 유지
   - 사용자가 필요할 때만 펼쳐서 확인하도록 유도
3) **요약형/확장형 모드**
   - 요약형: order/queue/stack만 표시
   - 확장형: dist/parent/trace 로그까지 표시

### 구현 제안
- `CTPPlayground`에 `showStatePanel`/`statePanelMode` 옵션 추가
- 모듈별 config에서 명시적으로 제어  
  예: DFS/BFS = `showStatePanel: true`, Sorting = `false`

### 적용 현황(이번 단계)
- `showStatePanel`/`statePanelMode` 옵션 도입
- DFS/BFS 관련 확장 모듈에 **선택적 노출 적용**
  - 기본은 summary 모드, 경로 복원은 full 모드
- Sorting 모듈은 **상태 패널 비활성화**(시각화 공간 확보)
- 비선형 Graph(DFS/BFS/Cycle/Shortest/MST) **상태 패널 기본 활성화**(summary)

## 거리(dist) 표시 개선 전략(간선 라벨)
### 문제
- dist/parent가 패널에만 표시되어 “왜 거리 갱신이 되었는지” 직관이 약함

### 전략
1) **간선 라벨에 dist/weight 표시**
   - `edge_relax`, `dist_update` 발생 시 해당 간선에 label 표시
2) **현재 갱신 대상 강조**
   - dist 업데이트된 노드의 라벨을 `id (d=xx)`로 표기(이미 적용)
3) **활성 간선 강조**
   - `edge_relax` 이벤트에 색상 강조 적용

### 결론
- 상태 패널은 **모듈별 선택적 노출**이 가장 효율적
- dist는 **간선 라벨/강조로 시각화 내부에서 설명**하는 방식이 학습 효과가 높음

### 적용 현황(이번 단계)
- `edge_relax` 이벤트에 따라 **간선 라벨(w/d)** 표시 적용

---

## 학습 효율 증강 기획: 간선 중심 dist + 상태 범례 + 애니메이션
### 1) dist 표시 방식 개선(노드 라벨 → 간선 라벨 중심)
문제: `d=...`를 노드 라벨에만 붙이면 “왜 이 경로가 선택됐는지”가 약하게 전달됨.

전략:
- **가중치(weight)**: 모든 간선에 고정 라벨로 표시 `w=...`
- **거리 갱신(dist update)**:
  - `edge_relax` 이벤트가 발생한 간선에 **임시 라벨** `d=...` 표시
  - 대상 노드 라벨은 **최종 dist만 유지** (시각적 과밀 방지)
- **갱신 지속 시간**: 2~3 step 유지 후 자동 소멸 (멘탈맵 유지)

### 2) 상태 범례(legend) 고정 노출
정렬처럼 “색의 의미”를 항상 안내:
- 노드: active / visited / found
- 간선: consider / relax / active
- 라벨: `w=...`(가중치), `d=...`(현재 갱신 거리)

UI 위치:
- 그래프 상단 오른쪽 “미니 범례” (접힘/펼침 가능)

### 3) 애니메이션 제안
- **간선 강조**: edge_relax 시 stroke 두께 증가 + 색상 펄스
- **노드 전이**: active→visited 전환 시 scale 1.05 → 1.0 (짧은 bounce)
- **라벨 강조**: dist_update 라벨은 페이드 인/아웃

### 4) 적용 대상
- Shortest Path (Dijkstra/Floyd/0-1 BFS)
- MST (edge_relax 강조)
- DFS/BFS는 간선 라벨 최소화(과밀 방지)

### 5) 적용 단계
1) **간선 라벨 규칙 정립** (w/d 분리)
2) **범례 컴포넌트 추가** (Graph 전용)
3) **애니메이션 클래스 추가** (SVG stroke/label)
4) **모듈별 적용 기준 확정** (Shortest Path > MST > DFS/BFS)

### 적용 현황(이번 단계)
- Graph SVG에 **범례 컴포넌트** 추가
- `edge_relax` 간선 두께/라벨 강조 적용
- 노드/간선 **상태 애니메이션** 적용
  - 노드: active(펄스), found(글로우), visited(페이드)
  - 간선: active/consider(대시 이동), relax(글로우)
- 상태 패널 **토글(온/오프)** UX 추가
- 상태 패널 섹션 확장(활성/비교/인덱스/경로/선택 간선/총 비용/결과 표시)
- 요약 모드에서도 **이벤트 로그 표시** (최근 N개)

## 이벤트 없는 챕터 보강 계획(필수 트레이스 최소 규약)
### 문제 정의
- `traceOnly` 모드에서 **트레이스 이벤트가 부족하면** 시각화가 빈 상태로 보임
- 일부 챕터는 **node/edge 상태 변화가 부족**하여 애니메이션이 체감되지 않음

### 최소 이벤트 규약(그래프 계열)
- 노드: `node_active`, `node_visit`, `node_finalize(완료/확정)`
- 간선: `edge_consider`, `edge_relax`(가중치/경로 갱신)
- 거리 갱신: `dist_update(id, dist, parent)`로 **부모 간선 표기** 가능하도록 확장

### 보강 대상(우선 순위)
1) **Graph-Advanced(Topological/MST)**: node_finalize/edge_active 명확화
2) **Shortest Path(Dijkstra)**: dist_update에 parent 추가
3) **DFS/BFS 확장 챕터**: active/visit/finalize 이벤트 누락 여부 점검

### 적용 단계
1) `traceOnly: true` 모듈 전수 점검 → 이벤트 누락 목록화
2) 각 챕터 `initialCode`에 최소 규약 이벤트 삽입
3) 시각화 재검증(깜빡임/공백/무표식 개선 확인)

## dist 표기 정책 개선(노드 라벨 → 간선 라벨 중심)
### 문제 정의
- 현재 dist가 노드 라벨에 붙어 **“왜 이 경로가 갱신됐는지”가 약함**

### 개선 전략
- **간선 라벨**에 `d=...` 표시(갱신된 경로를 “간선 위”에서 시각화)
- 노드 라벨은 **최종 dist만 유지**하거나 상태 패널로 이동

### 적용 플랜
1) `dist_update` 이벤트 스키마 확장: `parent` 또는 `from` 포함
2) Graph SVG: `dist_update`는 **부모 간선 라벨 갱신**으로 처리
3) BFS/DFS 계열은 `dist_update` 호출 시 parent 정보 전달
4) 노드 라벨에 붙는 `d=...` 제거(과밀 방지)

### 적용 현황(이번 단계)
- BFS/0-1 BFS/멀티소스/그래프 BFS/다익스트라의 `dist_update`에 parent 전달 적용
- Graph SVG에서 dist_update를 **간선 라벨(d=...)**로 표시하도록 전환
- DFS/BFS 계열에 `node_finalize` 보강(완료 애니메이션 활성화)
- 비선형 MST에 `node_finalize` 보강(선택 노드 확정 표시)

## 다각형 레이아웃 적용 범위(최단경로 알고리즘)
### 현황
- **비선형 Graph 개념 챕터**는 정다각형 레이아웃 적용 완료
- **Algorithms DFS/BFS**도 정다각형 적용 완료
- **Shortest Path(Dijkstra)**는 기본 레이아웃(그래프) 사용 중

### 적용 판단
- 작은 그래프(≤9~11 노드): 정다각형이 구조 파악에 유리
- 가중치/경로 강조가 핵심인 챕터는 **정다각형 + 간선 라벨**이 조합 최적

### 플랜
1) Dijkstra 시뮬레이터 `layoutMode: "polygon"` 적용
2) Floyd-Warshall은 **행렬 시각화 유지** (배치 변경 불필요)
3) Graph-Advanced(MST)도 **정다각형 우선** + 필요시 mst 전용 모드 병행

### 적용 현황(이번 단계)
- Algorithms Dijkstra `layoutMode: "polygon"` 적용

## 깜빡임/즉흥 전환 문제(학습 효율 저하) 진단 & 개선안
### 증상 요약
- 노드/간선이 **순간적으로 나타났다 사라지는** 듯 보임
- 단계별 의미(왜 방문했는지/큐·스택 상태)가 **시각적으로 남지 않음**
- 사용자가 커스텀 그래프를 넣으면 **직렬 배치**로 읽기 어려움

### 리서치 기반 인사이트
- 그래프 레이아웃은 **가독성 지표(겹침, 교차, 퍼짐)**에 크게 영향을 받음 citeturn0search2
- 순환/대칭 구조는 **원형(다각형) 배치**가 구조 파악에 유리함 citeturn0search6
- 동적 그래프에서 **멘탈맵(배치 안정성) 유지**가 학습과 이해에 중요함 citeturn0search4turn0search8
- 애니메이션은 일부 과제에서 정확도를 높이지만, **안정된 레이아웃과 요약 정보가 필수** citeturn0search8

### 개선 방향(CTP 적용 전략)
1) **레이아웃 안정화(멘탈맵 유지)**
   - 정다각형/원형 배치로 **노드 위치 고정**
   - 재실행 시 배치 불변(랜덤 배치 금지)
2) **상태 누적 표시**
   - 방문 순서, 큐/스택, dist/parent를 **항상 노출**
   - 그래프는 “상태 변화”가 아니라 **상태 누적 흐름**으로 인지
3) **이벤트 로그 타임라인**
   - trace 이벤트를 누적 기록해 **의사결정 과정을 추적**
4) **커스텀 그래프 대응**
   - 노드 수가 늘어나도 **다각형/원형 배치**로 균등 분산
   - 허브가 명확한 경우만 중앙 노드 배치(오해 방지)

### 적용 우선순위
- 1차: Graph 개념 챕터(Representation/DFS/BFS/Cycle) → polygon 고정
- 2차: Algorithms DFS/BFS → polygon 유지 + 상태 패널 강화
- 3차: MST/Shortest Path → 계층형/순서형 배치 유지

## 알고리즘 DFS/BFS 학습 타당성 점검
### 관찰 요약
- 시각화가 **보였다/안보였다** 하는 증상은
  - 이벤트 기반(traceOnly)일 때 **노드 집합이 단계마다 달라지기 때문**
  - 방문하지 않은 노드는 **그림에서 사라져 멘탈맵이 깨짐**

### 개선 적용(이번 단계)
- 알고리즘 DFS/BFS는 **traceOnly 해제 → 전체 그래프 고정 렌더링**
- 이벤트는 색상/라벨 변화만 적용하여 **구조는 유지**
- 상태 패널을 요약형으로 제한해 **시각화 공간 확보**

### 타당성 결론
- DFS/BFS는 **구조 유지 + 방문 순서 누적**이 핵심 학습 포인트이므로
  전체 그래프를 고정 렌더링하는 방식이 더 타당함

## CTP에 녹이는 구성 전략(결론)
### 핵심 판단
- “허브/루트가 있을 때만 다각형+중앙” 방식은 **코드 로직마다 달라질 수 있음**.
- 따라서 **자동 판별보다, 섹션별 ‘기본 예제/레이아웃’ 고정**이 학습 안정성을 높임.

### 적용 원칙
1) **섹션별 기본 레이아웃 고정**
   - Graph 개념: **원형(circular)** — 구조/노드 수 인지에 유리
   - DFS/BFS(알고리즘): **계층형(levels)** — 탐색 레벨/순서 시각화
   - MST/Shortest Path: **원형 + 강조 경로** (필요 시 force-lite 옵션)
   - Trie/Tree: **계층형 트리**
2) **정다각형+중앙은 ‘허브 예제’ 전용**
   - 스타 그래프/중앙 집중형 그래프 등 **명시된 예제에서만 사용**
3) **초기 코드(예제 데이터)로 학습 목표를 통제**
   - 예제 데이터 자체를 섹션 목표에 맞게 설계
   - “허브 구조 설명” 섹션만 허브 그래프를 제공
4) **사용자 커스텀은 ‘옵션’으로 열어두되**
   - 기본 학습 흐름은 **고정된 예제 + 고정 레이아웃**으로 보장

### CTP 실행 구조 제안
- `CTPModule`에서 `layoutMode`를 **명시적으로 지정**  
  (auto-detect 최소화)
- 개념별 예제 데이터셋을 **교과서형으로 통일**  
  (허브/그리드/사이클 등 유형별로 준비)

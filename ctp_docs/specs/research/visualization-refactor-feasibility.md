# 어려운 시각화(그래프/트라이/UF/알고리즘) 리팩토링 타당성 보고서

## 1) 문제 정의
- 증상: 코드 실행은 성공하나 시각화가 **간헐적으로 비어 있음**
- 재현: BFS/Advanced Graph 등 일부 세션에서 **처음 보였다가 재실행 시 영구적으로 비어짐**
- 영향: 학습 흐름이 끊기고, 코드→시각화 연결 신뢰도가 크게 저하됨

---

## 2) 기술적 원인 가설(관측 기반)
1) **시각화 엔진 타이밍 문제**
   - Cytoscape 초기화 이전에 요소 주입이 발생
   - 컨테이너 크기 0 상태에서 layout 수행 → 빈 화면 유지
2) **Adapter 출력 불안정**
   - globals 직렬화/변환 과정에서 그래프 구조 손실 가능
   - 변수명 규약 불일치 시 nodes/edges가 빈 배열
3) **Skulpt 호환성 한계**
   - `heapq` 등 미지원 모듈 사용 시 실행 중단
4) **재실행 시 상태 초기화 불완전**
   - 이전 cy 인스턴스/레이아웃/이벤트 상태가 남아 새 요소 주입 실패

---

## 3) 학습 효과 관점 리스크
- 고난이도 개념은 **시각화 신뢰성**이 핵심
- 실행 성공 → 빈 화면은 학습 몰입을 크게 훼손
- "코드→시각화"의 기대가 깨지면 전체 플랫폼 신뢰도 하락

---

## 4) 해결 전략 후보

### A) 현 구조 유지 + 안정화
**장점**
- 기존 아키텍처 재활용
- 빠른 패치/문서 업데이트 가능

**단점**
- 복잡한 그래프 레이아웃 문제는 근본 해결 어려움
- Skulpt/Adapter/시각화 간 결합도 높아 디버깅 비용 큼

**필수 작업**
- Cytoscape 렌더 타이밍 가드 강화
- Adapter 결과 로깅/검증 UI
- 재실행 시 cy 상태 강제 리셋

---

### B) Tracer 중심 구조로 완전 전환 (권장)
**핵심 아이디어**
- **코드 실행 → Trace 이벤트 → 상태 머신 → 렌더링**
- 시각화 엔진은 **Trace만 신뢰**하고, globals 기반 추론 최소화

**장점**
- 이벤트 기반이므로 디버깅 명확
- 복잡한 구조도 **상태 기반 렌더링** 가능
- 학습 효과 강화 (의도된 시각화만 표시)

**단점**
- 이벤트 템플릿 정의가 늘어남
- 초기 샘플 코드 제작 비용 증가

---

### C) 아예 새로운 렌더러(커스텀/캔버스) 도입
**장점**
- 시각화 스타일/레이아웃 완전 통제
- performance 최적화 가능

**단점**
- 개발 비용 큼
- 안정화까지 시간 소요

---

## 5) 타당성 결론
- **고난이도 세션은 Tracer 기반 구조로 전환하는 것이 타당**
  - 이유: 디버깅 가능성/안정성/학습 효과에서 가장 명확한 이득
- 단기적으로는 A(안정화)로 시급한 렌더 공백을 줄이고,
  중기적으로 B(Tracer 중심)으로 전환하는 **2단계 전략**이 현실적

---

## 6) 추천 실행 플랜(요약)
1) **단기(1주)**
   - Cytoscape 렌더 안정화 (레이아웃 재시도/강제 리셋)
   - Adapter 결과 로깅 + 디버그 오버레이
2) **중기(2~3주)**
   - Tracer 이벤트 기반 Graph/Trie/UF 시각화 확장
   - Adapter 의존 최소화 (Trace만으로 상태 결정)
3) **장기(선택)**
   - 필요 시 전용 렌더러(캔버스/커스텀) 검토
